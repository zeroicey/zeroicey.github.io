Harness Engineering 知识整理

发表于 2026-04-06 分类于 技术解析 本文字数： 3.5k 阅读时长 ≈ 3 分钟

Harness（马具/框架）是 AI 智能体中除了模型本身以外的所有东西。一个良好的 harness 服务于两个目标：提高智能体一次做对的概率，以及提供反馈循环帮助自我纠正。

Harness Engineering 知识整理

来源：Harness engineering for coding agent users - Martin Fowler

核心概念

Agent = Model + Harness

一个良好的外部 harness 服务于两个目标：

  1. 提高智能体一次做对的概率
  2. 提供反馈循环，在问题传到人眼之前尽可能自我纠正

两大控制机制

1. Feedforward（前馈控制）

  • 提前预测智能体不应该产生的输出，并尝试阻止
  • 在智能体行动之前引导其行为
  • 目的：提高智能体一次创建好结果的概率

2. Feedback（反馈控制）

  • 在智能体行动后观察结果
  • 帮助智能体进行自我纠正
  • 特别强大的是那些为 LLM 消费优化的信号，例如自定义 linter 消息包含自我纠正指令

关键洞察：只有反馈的智能体会重复同样的错误；只有前馈的智能体会制定规则但从不知道它们是否有效。

两种执行类型

类型 特点 示例
Computational（计算型） 确定性的、快速的，由 CPU 运行 测试、linter、类型检查、结构分析
Inferential（推断型） 语义分析、AI 代码审查，由 GPU 运行 LLM as judge

权衡：

  • Computational 控制：便宜、快速，可对每次变更运行
  • Inferential 控制：更贵、非确定性，但能提供更丰富的指导和额外的语义判断

三大监管类别

1. Maintainability Harness（可维护性 harness）

调节内部代码质量和可维护性。

例子：

  • 计算型传感器：重复代码、循环复杂度、缺失测试覆盖率、架构漂移、风格违规
  • 推断型传感器：语义重复代码、冗余测试、暴力修复、过度工程

局限性： 无法可靠地捕捉高影响问题，如：误诊问题、过度工程、不必要的功能、理解错误的指令。

2. Architecture Fitness Harness（架构适应性 harness）

定义和检查应用的架构特性（Fitness Functions）。

例子：

  • 前馈：描述性能要求的 Skills，性能测试
  • 反馈：检查是否改善或降低了性能要求

3. Behaviour Harness（行为 harness）

指导并感知应用的功能行为是否如预期工作。

当前做法：

  • 前馈：功能规范（从简短提示到多文件描述）
  • 反馈：AI 生成的测试套件是否通过、覆盖率、结合人工测试

这是最大的挑战——目前让智能体高自主工作的团队大多依赖 AI 生成的测试，质量还不够好。

Harnessability（可 harness 性）

不是每个代码库都同样容易使用 harness。特性：

  • 强类型语言：有类型检查作为传感器
  • 清晰的模块边界：支持架构约束规则
  • 框架抽象：隐藏智能体不必担心的细节

Ambient Affordances（环境可供性）：

“环境的结构属性，使其对在其中运行的智能体可读、可导航、可处理”

  • Greenfield：从第一天就可以内置可 harness 性
  • Legacy：技术债务多的应用最需要 harness，但也是最难构建的

关键原则

Ashby’s Law（必要多样性定律）

调节器必须至少有与其治理的系统一样多的多样性

LLM 智能体几乎可以生成任何东西，但承诺一种拓扑结构会缩小空间，使全面的 harness 更容易实现。

Timing: Keep Quality Left（质量左移）

尽可能在开发时间线的左侧（更早阶段）进行检查：

  • 什么应该在集成前甚至提交前运行？（linter、快速测试套件）
  • 什么更昂贵，应该只在流水线中运行？（突变测试、更全面的代码审查）

实践案例

OpenAI

  • 分层架构由自定义 linter 和结构测试强制执行
  • 定期”垃圾收集”扫描漂移并让智能体建议修复

Stripe

  • 基于启发式的相关 linter 预推送钩子
  • “蓝图”将反馈传感器集成到智能体工作流中

长时间运行智能体的 Harness 设计

基于 Effective harnesses for long-running agents

核心挑战

长时间运行的智能体面临两个主要问题：

  1. 一次做太多：智能体试图一次性完成整个应用，导致上下文耗尽
  2. 过早宣布胜利：在构建了一些功能后，后续的智能体会看到已经完成的进展，然后宣布工作完成

两部分解决方案

1. Initializer Agent（初始化智能体）

第一个智能体会话专门负责：

  • 设置初始环境（init.sh 脚本）
  • 创建 claude-progress.txt 进度日志文件
  • 创建详细的 feature list（功能列表）
  • 进行初始 git 提交

2. Coding Agent（编码智能体）

每个后续会话：

  • 读取进度文件和 git 历史了解状态
  • 每次只做一个功能（增量进展）
  • 结束时提交 git 并更新进度文件

关键组件

Feature List（功能列表）

使用结构化的 JSON 文件列出所有需要实现的功能：

规则：智能体只能修改 passes 字段，不能删除或修改测试。

启示

有效软件工程师每天做什么，我们就向智能体学习什么

  • 每次只做一个任务
  • 留下清晰的进度供下一个”工程师”阅读
  • 在开始新工作前验证基本功能仍然正常
  • 使用 git 回滚错误的代码更改

OpenAI 的 Harness Engineering 实践

基于 在智能体优先的世界中利用 Codex

核心成果

  • 5 个月构建 100 万行代码
  • 3 人团队 → 每天 3.5 个 PR
  • 零人工代码贡献 - 所有代码由 Codex 编写

关键经验

1. 从空仓库开始

  • 初始架构由 Codex 使用 GPT-5 生成
  • AGENTS.md 也是由 Codex 编写
  • 完全没有预存人工代码

2. 重新定义工程师角色

人类掌舵，智能体执行

工程师的工作不再是写代码，而是：

  • 设计环境
  • 明确意图
  • 构建反馈回路

3. 给智能体”地图”而非”说明书”

❌ 失败的尝试：一个巨大的 AGENTS.md（1000页手册）

  • 情境是稀缺资源
  • 过多指导变得无效
  • 文档会快速腐烂
  • 难以验证

✅ 正确做法：AGENTS.md 作为”内容目录”（约100行）

渐进式披露：从一个小而稳定的切入点开始，智能体被引导下一步该去哪里查看。

4. 目标是智能体的可读性

从智能体的角度看：运行时无法访问的任何内容都不存在

  • 存储在 Google Docs、Slack、人类头脑中的知识都无法被访问
  • 代码仓库本地的、版本化的工件是智能体能看到的全部
  • 需要将情境推送到仓库中

5. 通过约束而非微观管理

  • 严格边界和可预测结构是关键
  • 自定义 linter 强制执行架构规则
  • 结构测试验证依赖方向

6. 熵与垃圾收集

智能体会复制现有模式（包括不理想的模式），导致漂移。

解决方案：”黄金原则” + 定期清理

  • 将品味编码为机械规则
  • 定期运行后台任务扫描偏差
  • 技术债务就像高息贷款：小额定期偿还 > 痛苦一次性解决

核心洞察

构建软件仍然需要纪律，但纪律更多地体现在支撑结构上，而不是代码上。

最棘手的挑战集中在：

  • 设计环境
  • 反馈回路
  • 控制系统
